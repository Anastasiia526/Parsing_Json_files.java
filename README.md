Цей проект реалізує зв'язок багато-до-одного на прикладі "багато-фільмів-одного-режисера".
Реалізується парсинг JSON-файлів з інформацією про фільми, збереження їх у форматі JSON по режисерах, та генерацію статистики у форматі XML за різними атрибутами фільмів (жанр, рік, назва).
Також реалізовано багатопотоковий парсинг для пришвидшення обробки великої кількості файлів.
Основні сутності
1. Movie - представляє фільм:

title – назва фільму

director – ім’я режисера

yearOfRelease – рік випуску

genre – список жанрів (List<String>)

2. WriteToJsonFiles - клас для:

Додавання фільмів у колекцію по режисерах

Збереження колекції у JSON-файли

3. ReadByJsonFiles - клас для:

Читання JSON-файлів з фільмами

Генерації статистики за атрибутами (title, genre, year)

Запис статистики у XML-файл

4. ExecutorThreadJsonParser - клас для:

Паралельного парсингу JSON-файлів

Вимірювання часу виконання при різній кількості потоків

Об’єднання результатів парсингу у єдину колекцію

5. Приклад вхідного JSON

Файл Clint_Eastwood.json:

{
  "Clint Eastwood": [
    {
      "title": "Play Misty for Me",
      "yearOfRelease": 1971,
      "genre": ["Thriller"]
    },
    {
      "title": "Bronco Billy",
      "yearOfRelease": 1980,
      "genre": ["Comedy", "Drama"]
    }
  ]
}

6. Приклад вихідного XML (статистика за жанром)
<statistics>
    <item>
        <name>Comedy</name>
        <count>40</count>
    </item>
    <item>
       <name>Drama</name>
        <count>87</count>
    </item>
    <item>
        <name>Thriller</name>
        <count>18</count>
    </item>
</statistics>

7. Приклади запуску багатопотокового парсингу

Клас ExecutorThreadJsonParser дозволяє протестувати обробку файлів з різною кількістю потоків:

Threads: 2  | Time: 12 ms | Directors parsed: 5
Threads: 4  | Time: 7 ms  | Directors parsed: 5
Threads: 8  | Time: 5 ms  | Directors parsed: 5
Threads: 10 | Time: 4 ms  | Directors parsed: 5
Багатопотоковий підхід суттєво прискорює парсинг JSON-файлів, але ефективність зростає лише до певної межі. Оптимальний результат досягається при 4–8 потоках, далі продуктивність майже не покращується.



Висновок:
Збільшення кількості потоків зменшує час парсингу, але після певного числа потоків (8–10) ефект від багатопоточності слабшає через накладні витрати на управління потоками.
